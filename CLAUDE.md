## What this is

Igor Widlinski's personal portfolio + blog (Hugo static site), live at **https://widlin.ski**. Theme is [Toha v4](https://github.com/hugo-toha/toha) pulled in as a **Hugo module** (see `go.mod`) ŌĆö it is not vendored, so theme templates live in the Hugo module cache, not this repo.

## Deploys ŌĆö pushing to main goes live

Every push to `main` publishes the live site via **two** parallel paths:

- **Netlify** (`netlify.toml`) ŌĆö production runs `hugo --gc --minify`
- **GitHub Pages** (`.github/workflows/hugo.yaml`) ŌĆö runs `npm ci` then `hugo --gc --minify`, so `package-lock.json` must stay committed and in sync with `package.json`

A `.gitlab-ci.yml` also exists (GitLab Pages); unclear if still active. For risky changes, prefer a branch ŌĆö Netlify builds deploy previews.

## Building locally ŌĆö Hugo version matters

The build is **pinned to Hugo extended 0.163.2** (in `netlify.toml` and the GitHub workflow). Other Hugo versions can fail or behave differently against the pinned Toha version ŌĆö do not trust a system-installed hugo. Newer Hugo releases ship macOS builds only as `.pkg`; extract the binary instead of installing:

```bash
curl -sL https://github.com/gohugoio/hugo/releases/download/v0.163.2/hugo_extended_0.163.2_darwin-universal.pkg -o /tmp/hugo.pkg
pkgutil --expand-full /tmp/hugo.pkg /tmp/hugo_pkg   # binary at /tmp/hugo_pkg/Payload/hugo

npm install                                # once, or when package.json changes
/tmp/hugo_pkg/Payload/hugo --gc --minify   # build into public/
/tmp/hugo_pkg/Payload/hugo server          # local dev server
```

There are no tests or lint for site content; verification = a clean `hugo --gc --minify` (exit 0, no WARN/ERROR) plus spot-checking `public/`.

## Upgrading Hugo / Toha

- Bump the theme: `hugo mod get github.com/hugo-toha/toha/v4@<version> && hugo mod tidy`
- Update `HUGO_VERSION` pins in **all four** contexts of `netlify.toml` and in `.github/workflows/hugo.yaml`
- Keep the root `package.json` devDependencies aligned with the theme's `package.hugo.json` (in the module cache). Do **not** rely on `hugo mod npm pack`: on newer Hugo it generates a `packages/hugoautogen` npm workspace (gitignored) instead of updating the root file, and mismatched root pins shadow the theme's deps and break the esbuild JS bundle.
- Newer Hugo is strict: duplicate YAML config keys are hard errors; `languageCode`/`languageName` are deprecated in favor of `locale`/`label`.

## Architecture

- **`hugo.yaml`** ŌĆö site config; theme features toggle under `params.features` (notes and comments are disabled; portfolio/blog enabled)
- **`data/en/`** ŌĆö drives the homepage. `author.yaml` is the hero section; `sections/*.yaml` are the homepage sections, each with a `section.enable` flag and `weight` ordering. **Note:** `education`, `projects`, `skills`, `publications`, `achievements`, `accomplishments` still contain Toha's demo/lorem-ipsum data and are deliberately kept `enable: false` as scaffolding ŌĆö don't mistake them for real data, and don't enable them without replacing the content. `about.yaml` and `experiences.yaml` are real.
- **`data/en/sections/featured-posts.yaml`** ŌĆö lists posts by path (e.g. `/posts/aws/2025/sap-c02`), resolved via `site.GetPage`; a bad path silently renders nothing
- **`content/posts/<category>/<year>/<slug>/index.md`** ŌĆö page bundles with images alongside. Front matter uses Toha's sidebar menu: `menu.sidebar` with an `identifier` and a `parent` matching a category `_index.md` identifier (e.g. `AWS-2025`). `archetypes/default.md` defaults new posts to the AWS-2025 parent with `draft: true`.
- **`static/CNAME`** ŌĆö the custom domain for GitHub Pages; don't remove
- **`assets/jsconfig.json`** ŌĆö auto-generated by Hugo, points into the module cache; its diff churn on theme upgrades is expected
